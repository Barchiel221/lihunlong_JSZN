## 第一部分：四个区域方案确定

### 0. 决策总表

| 区域 | 方案结论 | 需要的新能力 | 代码改动量 |
|---|---|---|---|
| ① 立柱阵 | **现有栈基线方案直接飞**（省赛已验证），只做 220mm 参数重整 | 无 | 仅调参 |
| ② 树林+风扇 | **限速绕行方案**：航段 max_vel 0.5，航点布在背风侧，抗风交给 PX4 内环 | 航段级限速（F2） | 参数切换机制 |
| ③ 移动障碍 | **高频重规划+残影清除方案**：不做预测，收小 local_update_range 加快 raycast 清障，路线与障碍运动方向垂直快速通过 | 无（纯调参+航点布置） | 仅调参 |
| ④ 0.7m 拱门 | **引导航点低速直穿方案**：拱门前后法向 ±0.8m 各设引导航点，航段 max_vel 0.4；若环④低位，该航段单独下调 virtual_ground | 航段级限速 + 航段级 ceil/ground（F2） | 参数切换机制 |
| 全程 | **mission.yaml 机载自动任务调度**（规则禁止依赖 RViz 点 goal） | 任务执行器（F1） | 新节点 |
| 全程 | **机头跟随飞行方向**（修复省赛锁 yaw 导致区域③④倒飞） | yaw 跟随器（F4） | offboard 改 ~25 行 |

四个区域**都不需要改 planner 算法**（EGO v1 无 ESDF、无动态预测，这是已知事实，方案全部绕开算法短板）。改动收敛为四件事：

- **F1 任务执行器**：读 mission.yaml 顺序发 goal（替代 RViz），支持"穿环不悬停"。
- **F2 航段级参数切换**：max_vel / virtual_ceil / virtual_ground 随航段切换（计划书 §4.2 认定的唯一新功能）。
- **F3 参数重整**：330mm→220mm 体积参数 + 场地适配参数，纯 launch 改动。
- **F4 yaw 跟随器**：offboard 转发段把机头从"锁死起飞朝向"改为"速度死区 + 限速跟随飞行方向"，消除区域②→③右转 90° 后的倒飞，同时不复活省赛的起飞猛甩（详见改动 I）。

### 1. 区域①（立柱阵，8–10 根 0.5×0.5×2m）——基线方案

**选型理由**：与省赛场景同类，现有栈（inflated occupancy + A* + B 样条优化）已实飞验证；立柱是结构化障碍，点云清晰，无新风险。

- 参数：220mm 重整后的基线档（`obstacles_inflation=0.20`，`max_vel=0.6`），不做航段特殊化。
- 航点：环①中心 1 个 goal（若环口方向已知，前后加引导航点垂直穿环）。
- **对代码的需求：无**（F1/F3 覆盖）。

### 2. 区域②（5–7 棵 ≥1.8m 假树 + 角落风扇）——限速绕行方案

**选型理由**：树高 1.8m 贴近 2m 网顶，翻越无余量，只能绕。树冠点云稀碎但 inflation 0.20 + dist0 0.08 足以兜住 voxel 级碎点；风扰是**跟踪误差**问题而非规划问题，对策是降低本段速度让 PX4 内环有姿态余量，规划器本身不改。

- 航段参数：`max_vel=0.5`（本段），其余不变。
- 航点：环②goal 布在**背风侧**（远离风扇角落），必要时加 1 个中间航点把路线整体拉离风扇。
- 明确不做的：不改 lambda 权重、不做风估计/前馈（收益不确定、验证成本高，违背"稳>快"）。
- **对代码的需求：F2 航段级 max_vel**。

### 3. 区域③（2 个 0.5×0.5×2m 移动障碍，≤0.5m/s）——高频重规划+残影清除方案

**选型理由**：EGO v1 无动态障碍预测，加预测=改算法=高风险。障碍仅 0.5m/s，而 replan 周期远快于障碍位移一个 voxel 的时间；瓶颈是**残影**（障碍旧位置的占据没被及时 raycast 清掉，把走廊堵死）。因此方案 = 让地图更新更快更干净 + 用路线几何降低相遇概率。

- 参数：`local_update_range_x/y` 8.0→**6.0**（减少每周期处理量、缩短残影存活时间；仍 ≥ planning_horizon 4.0 + 1）。
- 航点：进出区域③的两个航点连线**与障碍运动路径垂直**，缩短暴露时间；本段维持 0.6 不减速（快速通过比慢速躲避更优——障碍高 2m，从上方过不可行）。
- 备选（实测残影仍堵路才启用）：调 `fsm/thresh_replan_time` 提高重规划灵敏度——放到 M5 演练实测后决定，不进首版。
- **对代码的需求：无**（F3 覆盖 local_update_range；航点几何写进 mission.yaml）。

### 4. 区域④（0.7m 拱门）——引导航点低速直穿方案

**选型理由**：计划书 §4.1 验算——220mm（R≈0.15）+ inflation 0.20 后，0.7m 开口剩 0.30m 自由走廊，机心可用 ±0.07m，可行但没有挥霍空间。因此必须：对正开口、直线、低速、**禁止在洞口内触发转弯重规划**。

- 航点：拱门开口中心前后**法向 ±0.8m** 各 1 个引导航点（穿环同理），保证进入洞口时轨迹已经是直线。
- 航段参数：`max_vel=0.4`（本段）。
- 低位环风险：若公布的环④中心 <0.7m，与全局 `virtual_ground 0.45/enable 0.55` 冲突（planner 拒绝下到环高），该航段需把 ground/enable 切到 **0.25/0.35**，过环后切回。
- **对代码的需求：F2 航段级 max_vel + 航段级 virtual_ground/ceil**。

### 5. 全程调度——mission.yaml 机载任务方案

**选型理由**：规则"全程自主 + 路由器受控"，goal 不能靠人在 RViz 点。现有 `multi_waypoint_publisher.py` 已实现顺序发 goal + 到点判定 + 最后一点交给 px4_offboard 走 FINAL_CORRECTION→LAND→DISARM 链路，**扩展它而不是新写**。

两处能力缺口：

1. **穿环不悬停**：现有到点判定要求 `v ≤ 0.20 m/s` 且 settle 1s（multi_waypoint_publisher.py:116-123），会在每个环前近似悬停。环航点需要 fly-through 模式：进入 XY 容差即切下一点，不查速度不 settle。
2. **航段参数下发**：切航段时把该段 max_vel/ceil/ground 推给 planner（依赖 F2 的动态参数回调）。

与 px4_offboard 的交互已确认安全：offboard_node.cpp:224-226，FINAL_CORRECTION/HOLD 状态下收到新 goal 会回退 FLY，且 `hold_seconds=5.0` 提供窗口；fly-through 模式下 sequencer 必然先于 offboard 的 1s settle 发出下一 goal，不会误触发降落链路。

### 改动 A：grid_map 运行时参数回调

现状：`grid_map.cpp:81-85` 启动时一次性 `get_parameter` 读入 `mp_.virtual_ceil_height_` / `mp_.virtual_ground_height_` / `mp_.virtual_ground_enable_height_`，之后不再读。

改法：在 `GridMap::initMap` 末尾注册 `node_->add_on_set_parameters_callback`，只处理这三个 key，更新对应 `mp_` 成员并打 INFO 日志；其余参数一律 `successful=true` 直接放行（不干扰同节点其他回调）。

**可行性已确认**：ceil/ground 的占据填充在 `clearAndInflateLocalMap()`（grid_map.cpp:290-310 一带）内进行，而该函数每次占据更新都重建局部 inflate 层（grid_map.cpp:832-833），因此成员变量更新后**下一个地图更新周期自动生效，无残留脏 voxel**。

约束检查保留：回调里复用 `grid_map.cpp:93-95` 的逻辑（ceil 不得超过 map z 上界），非法值拒绝并返回 `successful=false`。

### 改动 B：max_vel 运行时更新

现状：两处独立持有 max_vel，都是启动时读一次：
- `planner_manager.cpp:24` → `pp_.max_vel_`（用于分配轨迹时长，planner_manager.cpp:75/95/298/447/514 全部即时读 `pp_`，成员更新即生效）；
- `bspline_optimizer.cpp:34` → `max_vel_`（feasibility 代价，bspline_optimizer.cpp:1147 起）。

改法：
1. `BsplineOptimizer` 加公开 setter `setMaxVelDynamic(double)`（成员是 private）。
2. 在 `EGOReplanFSM::init`（能同时拿到 planner_manager 与 node）注册第二个 `add_on_set_parameters_callback`，处理 `manager/max_vel` 与 `optimization/max_vel`：更新 `pp_.max_vel_` 并调用 optimizer setter。ROS 2 允许同一节点链式注册多个回调，与改动 A 不冲突。
3. **只允许下调后再回调不超过 launch 初值**（0.6 保守档）：回调里以启动值为上限做 clamp。理由：`emergency_time=1.0` 与 `planning_horizon=4.0` 都是按 max_vel≤0.6 标定的，航段限速只会更慢、约束只会更安全；提速档（0.8–1.0）不走运行时通道，走 launch 档位（改动 E），届时 emergency_time/planning_horizon 一并按档位调整。
4. max_acc 不做运行时切换（无航段需求，少一个变量）。

### 改动 C：任务执行器 mission_executor

在 `multi_waypoint_publisher.py` 基础上扩展（保留原命令行模式向后兼容，或新建 `mission_executor.py` 复用其类），新增：

1. **`--mission <yaml>` 输入**，схема见改动 D。
2. **fly_through 到点判定**：waypoint 带 `fly_through: true` 时，判定退化为 `d_xy <= xy_tol`（不查 z 速度、不 settle），穿环航点与引导航点全用它；普通航点维持现有三条件判定。
3. **航段参数下发**：waypoint 可带 `segment_params:`（max_vel / virtual_ceil_height / virtual_ground_height / virtual_ground_enable_height 的任意子集）。发布该 goal **之前**，通过 `/drone_0_ego_planner_node/set_parameters` 服务（rclpy `SetParameters`）同步推送，**等到服务应答成功再发 goal**（保证 replan 用的是新约束）。key 映射：`max_vel` → 同时设 `manager/max_vel` 与 `optimization/max_vel`。
4. **失败策略**：参数服务 3 次重试（每次 1s 超时）仍失败 → ERROR 日志 + **继续任务但沿用旧参数**（旧参数是更保守的一侧时安全；若该段是低位环④、旧 virtual_ground 会挡住下降，则 EGO 到不了点 → 最终由 px4_offboard 状态超时 LAND 兜底，不会失控）。此策略在 dry-run 中专项验证。
5. **最后一个航点语义不变**：发出后 sequencer 退出判定，交 px4_offboard 走 FINAL_CORRECTION（0.10/0.15）→ LAND → DISARM，该链路已实飞验证、不动。

不改 px4_offboard_pkg：goal 抢占逻辑（offboard_node.cpp:224）、精降链路、超时兜底均满足需求。

### 改动 I：offboard_node yaw 跟随（消除倒飞）

**背景**：省赛为压掉"起飞后机头猛甩约 90°"，把 FLY 转发段的 yaw 硬锁成 `home_yaw_`（`offboard_node.cpp:530-531`），全程平移不转头。国赛地图区域②→③需右转 90°，锁死后区域③④将**倒着飞**。需在不复活起飞猛甩的前提下让机头跟随飞行方向。评分不含朝向（起飞 10+4 环×18+降落 10+碰撞 8），因此本改动纯为消除倒飞这一行为诉求；MID360 360° 水平视野下倒飞本不影响避障，故方案只需"看起来对、且稳"，不承担避障功能。

**根因**：EGO traj_server 的期望 yaw 取路径切向 `atan2(dir_y,dir_x)`（`traj_server.cpp:81`），而其内部 `last_yaw_` 全局初值为 0（`traj_server.cpp:24`）、并非飞机真实起飞朝向 → 起飞第一条轨迹一生成，yaw 命令即从 0 跳向切向，PX4 以最大速率猛甩。锁死是把这条 yaw 完全挡在 PX4 之外。

**方案**：不动 traj_server（上游 fork + replay 同步约束），改在 offboard 转发段自算 yaw。offboard 内一切为 NED，`atan2(vy_ned,vx_ned)` 直接即 NED 机头角，无需任何 ENU↔NED 翻转（比信任 adapter 转换过的 `msg->yaw` 更少出错面）。三件事：

1. **数据齐备已确认**：`ext_sp_.velocity[0]/[1]`（NED North/East，adapter 默认 `publish_velocity=true` 填充，`ego_px4_adapter_node.cpp:56-58`）、`pos_.heading`（PX4 回报真实机头）、`home_yaw_`（起飞朝向，`offboard_node.cpp:463`）、定时器周期 `1.0/rate_hz`（`offboard_node.cpp:176`）。
2. **替换 `offboard_node.cpp:528-531`** 的 `sp.yaw = home_yaw_`：

   ```cpp
   auto sp = ext_sp_;
   sp.timestamp = now_us();
   // 从 EGO 速度方向算期望机头（全在 NED，无坐标翻转）
   const double vx = sp.velocity[0], vy = sp.velocity[1];
   const double speed = std::hypot(vx, vy);
   double yaw_des = cmd_yaw_;                              // 默认保持
   if (std::isfinite(speed) && speed > yaw_follow_speed_)  // 速度死区，默认 0.3 m/s
       yaw_des = std::atan2(vy, vx);
   // 转向限速：一步最多转 yaw_rate_max_ * dt
   const double dt = 1.0 / rate_hz_;
   double dyaw = wrap_pi(yaw_des - cmd_yaw_);
   dyaw = std::clamp(dyaw, -yaw_rate_max_ * dt, yaw_rate_max_ * dt);  // 默认 60°/s
   cmd_yaw_ = wrap_pi(cmd_yaw_ + dyaw);
   sp.yaw = static_cast<float>(cmd_yaw_);
   sp.yawspeed = NAN;
   pub_sp_->publish(sp);
   ```

3. **种子**：TAKEOFF→FLY 切换处（`offboard_node.cpp:519` 附近）令 `cmd_yaw_ = home_yaw_;`。这是消除起飞猛甩的根本——从飞机**真实**朝向起步，起飞/悬停速度落在死区内则 `cmd_yaw_` 保持不动、无阶跃。
4. **配套**：新增成员 `cmd_yaw_`、复用 adapter 里的 `wrap_pi`；新增两参数 `yaw_follow_speed`（默认 0.3）、`yaw_rate_max_deg`（默认 60，构造时转 `yaw_rate_max_` 弧度/秒）；`rate_hz_` 若未存成员则一并补。参数写进改动 E 的 launch，三档位默认相同。

**行为对照**：起飞/悬停→死区内保持 home_yaw_ 不 snap；区域②→③右转 90°→机头以 ≤60°/s 跟过去约 1.5s，B 样条大曲率处本就减速，呈自然转弯而非倒飞；穿环/拱门→飞机沿环法向或洞口直线运动（改动 D 的 ±0.8m 引导航点保证），机头自然正对开口垂直穿越，洞内无重规划转弯故 yaw 稳定。

**不改**：traj_server、状态机骨架/转换/精降链路/goal 抢占均不动，仅替换 FLY 转发段的 yaw 赋值（属自研集成代码，非"状态机主体"）。

**风险**：限速期间机头滞后速度方向，高速急弯会斜飞一小段 → `yaw_rate_max` 勿调过小（60°/s 对 max_vel 0.6 足够），或让急弯航段降速（区域②④本就限速）。**必须先 dry-run 录 bag**，画 `pos_.heading` vs 命令 yaw，确认起飞段无阶跃、转角段单调过渡再上真飞。

## 第四部分：仿真器验证（路线 B 衔接）

> 依据：`空中挑战赛_Unity仿真器_ROS2_PX4全栈接入指南_路线B.md`（仓库根目录）。
> 定位：**里程碑 1（真值定位）是主战场**——真值 odom + Unity 点云 + PX4 SITL 闭环，
> 恰好覆盖本方案改动所在的 goal 调度 → EGO 规划 → adapter → offboard → PX4 全链路。
> 它比 dry-run（不解锁、状态机走不完）和 bag 回放（开环）都强一档，是赛前唯一能
> 反复跑"完整任务闭环"的手段。**不建议做里程碑 2**：FAST-LIO 的验证价值在真机
> 振动/外参上，仿真里标不出来，成本（补 IMU 源、单位、时序）不值。

|---|---|---|
| A grid_map 动态参数 | ✅ 完整 | Unity 点云喂 grid_map，运行时切参后看 ceil marker / 规划带变化，与真机行为同构 |

### 仿真验证不了的（边界，别外推）

1. **FAST-LIO 与改机相关的一切**（外参、振动、漂移）→ 只能走 M2 手持 bag + 悬停实飞。
2. **220mm 动力学与风扰**：SITL 机型不是 220mm 机架；附录 B 的 Gazebo wind 插件只能测"扰动下状态机不异常"，测不了真实姿态余量，区域②限速值最终实飞定。
3. **碰撞与精降精度**：障碍无物理（红线 2），按轨迹清障判定；精确降落 10 分的表现是真机 EKF2 + 地效的事。
4. **算力**：仿真跑在 x86+GPU 宿主机，Orange Pi 5 上的 planner/建图负载单独评估。

### 衔接改动（并入第二部分工作量）

- **改动 H：`ego_sim_flight.launch.py`**（指南第 7 节的 sim 变体）：去掉 `lihunlong_run_launch.py` 传感层，`relay_cloud` 改订 Unity 点云 `/drone_0_pcl_render_node/cloud`，加 pose_relay（PUBLISH_LIO=True），planner 参数段与 ego_real_flight 一致，mission_executor 按 `mission_file` 参数拉起。
- **参数段三份拷贝问题**：ego_real_flight / ego_replay / ego_sim_flight 三个 launch 各持一份 planner 参数，同步纪律迟早出错 → **把 `ego_advanced` 参数段抽成共享 Python dict**（如 `top_launch_pkg/launch/ego_planner_params.py`，三个 launch import 后按需覆盖），一处改三处生效。此项在 S2 之前做，替代原"人工双向同步"约定。
- 指南第 9 节"仓库里没有 waypoint 序列调度器、需新写适配节点"已过时（`multi_waypoint_publisher.py` 存在）；**改动 C 的 mission_executor 就是该节点**——仿真里读同一份 mission.yaml 逐点发 goal，真机原样复用。仿真验证的就是比赛当天跑的代码。
- 前提核查（指南第 3 节红线）：px4_msgs 与 PX4 SITL 固件同源（`/fmu/out/vehicle_local_position_v1` 等话题名一字不差），否则 offboard 卡 INIT；pose_relay 坐标符号在 rviz 现场标定（map 是 NWU）。

## 不改的东西（负面清单）

- FAST-LIO / lio_px4_bridge / ego_px4_adapter / px4_offboard 状态机主体（已实飞验证）。改动 I 仅替换 offboard FLY 转发段的 yaw 赋值，不动状态机转换、精降链路、goal 抢占。
- planner 算法本体：不加动态障碍预测、不加 ESDF、不改 lambda 权重、不做风前馈。
- `use_distinctive_trajs=False`（SIGSEGV bug）。
- traj_server、ENU↔NED 转换两处、identity static TF 桥。
